# Dashboard Core Integration for UI Components (Post-Rollback)

## Context

This document outlines the **current, simplified integration pattern** between UI components in `ui-terminal` and the `dashboard-core`, specifically how the UI consumes data provided by the `DashboardService` trait implementation.

- When building UI components that display dashboard data.
- When understanding how the `ui-terminal` fetches and displays data.

## Current Integration Pattern

The current integration relies on the `DashboardService` trait defined in `dashboard-core`.

1.  **Provider Injection:**
    *   The `ui-terminal` application (`app.rs::App`) receives an `Arc<dyn DashboardService + Send + Sync + 'static>` upon creation.

    ```rust
    // In lib.rs during app creation
    let app = app::App::new(provider.clone());
    ```

2.  **Data Fetching:**
    *   The `app.update()` method is called periodically.
    *   It calls `provider.get_dashboard_data().await` to fetch the consolidated `DashboardData` structure.

    ```rust
    // In app.rs
    async fn update(&mut self) {
        match self.provider.get_dashboard_data().await {
            Ok(dashboard_data) => {
                // Update AppState using dashboard_data
                self.state.metrics = Some(dashboard_data.metrics);
                self.state.alerts = dashboard_data.alerts;
                // ... derive connection status, update histories ...
            }
            Err(e) => {
                // Log error, update AppState error indicators
            }
        }
        self.state.last_update = Some(Utc::now());
    }
    ```

3.  **State Update:**
    *   The `AppState` struct stores the relevant pieces of the fetched `DashboardData` (or data derived from it).
    *   This includes `metrics`, `alerts`, derived `connection_status`, `connection_health` string, and specific time series like `cpu_history` and `memory_history`.

4.  **Widget Rendering:**
    *   UI rendering functions (in `ui.rs`) pass relevant parts of the `AppState` to individual widget rendering functions (in `widgets/*.rs`).
    *   Widgets are data-driven, rendering based on the provided state.

    ```rust
    // Example in ui.rs
    fn render_overview_tab<B: Backend>(app: &App, frame: &mut Frame<'_>, area: Rect) {
        // ... layout ...

        // Pass derived health checks to HealthWidget
        let health_checks = app.get_health_checks();
        widgets::health::render::<B>(frame, left_chunks[0], &health_checks);

        // Pass App reference to MetricsWidget (which accesses app.state.metrics)
        widgets::metrics::render::<B>(frame, app, left_chunks[1]);

        // Pass specific history VecDeques to ChartWidget
        widgets::chart::render::<B>(
            frame,
            right_chunks[0],
            "CPU Usage (%)",
            &app.state.cpu_history,
        );
        widgets::chart::render::<B>(
            frame,
            right_chunks[1],
            "Memory Usage (%)",
            &app.state.memory_history,
        );
    }
    ```

## Implemented Widgets and Data Flow

*   **`HealthWidget`:** Renders `Vec<HealthCheck>` generated by `app.get_health_checks()` based on `AppState.connection_status` and `AppState.metrics`.
*   **`MetricsWidget`:** Renders basic system metrics directly from `AppState.metrics`.
*   **`ChartWidget`:** Renders time series data passed directly from `AppState.cpu_history` or `AppState.memory_history`.

## Placeholder/Missing Widgets

*   **`NetworkWidget`:** Placeholder exists. Intended to render `AppState.metrics.network`.
*   **`AlertsWidget`:** Placeholder exists. Intended to render `AppState.alerts`.
*   **`ConnectionHealthWidget`:** Placeholder exists. Intended to render basic status derived from `AppState.connection_status` and `AppState.connection_health` string.
*   **`ProtocolWidget`:** Not yet created. Intended to render data derived from `DashboardData.protocol` (passed via `AppState`).

## Error Handling

*   Errors during `get_dashboard_data` are caught in `app.update()`, logged, and stored in `AppState.recent_errors`. The UI reflects this via the footer status bar and potentially resetting some displayed data.

## Testing

*   Testing should involve mocking the `DashboardService` trait to provide controlled `DashboardData` responses.
*   Unit tests for widgets should verify rendering based on specific `AppState` slices or derived data structures.
*   Integration tests can verify that `app.update()` correctly modifies `AppState` based on mock `DashboardData`.

## Future Considerations (Beyond Current Scope)

*   Direct MCP Integration (potentially bypassing `DashboardService` for some data).
*   More complex state management (e.g., historical data aggregation beyond simple VecDeques).
*   Advanced widget features (theming, configuration builders).
*   Lazy loading or more granular data fetching (instead of single `DashboardData`).
