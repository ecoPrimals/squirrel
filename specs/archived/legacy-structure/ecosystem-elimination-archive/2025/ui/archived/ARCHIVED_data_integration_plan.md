# Dashboard Core Data Integration Plan

## Goal

Replace dummy data generation within `crates/dashboard-core/src/service.rs` with real data fetched from monitoring sources, assumed to be accessible via the Machine Context Protocol (MCP) layer using an `McpClient` trait defined in `crates/dashboard-core/src/mcp.rs`.

## Current State

- `dashboard-core`'s `DefaultDashboardService` uses a background task (`start`) that periodically calls `collect_dashboard_data`.
- `collect_dashboard_data` currently uses `sysinfo` for minimal CPU data but relies on hardcoded dummy values for most metrics, protocol status, and alerts. It does not interact with MCP.
- The service broadcasts updates via a `tokio::sync::broadcast` channel.
- An `McpClient` trait and `MockMcpClient` exist in `dashboard-core/src/mcp.rs` but are not used by `DefaultDashboardService`.
- The public `update_dashboard_data` method exists, suggesting an alternative push-based update path is also possible.

## Proposed Integration Strategy (Polling via McpClient)

This plan focuses on integrating data fetching *within* the existing `collect_dashboard_data` loop using the `McpClient` trait.

**Phase 1: Service Modification & Mock Integration**

1.  **Add `McpClient` to `DefaultDashboardService`:**
    *   Modify the `DefaultDashboardService` struct definition in `service.rs` to include a field like `mcp_client: Arc<Mutex<dyn McpClient>>`.
2.  **Inject `McpClient`:**
    *   Update `DefaultDashboardService::new` and `::default` to accept or create an `McpClient` instance (e.g., `Arc::new(Mutex::new(MockMcpClient::new()))` for initial implementation/testing) and store it in the new field.
3.  **Modify `collect_dashboard_data`:**
    *   Remove all dummy data generation for metrics, protocol, and alerts.
    *   Remove direct `sysinfo` usage (unless `McpClient` doesn't provide equivalent system metrics).
    *   Inside the function:
        *   Acquire a lock on `self.mcp_client`.
        *   Call `mcp_client.get_metrics().await?`.
        *   Call `mcp_client.get_protocol_data().await?`.
        *   Call `mcp_client.get_alerts().await?`.
        *   Use the results from these calls to update the corresponding fields (`metrics`, `protocol`, `alerts`) in the service's shared `data` (`let mut data = self.data.write().await;`).
        *   Keep the existing logic for updating `timestamp` and `history` (using the *real* metrics data fetched via `McpClient`).
        *   Keep the history trimming logic.
4.  **Testing:**
    *   Run the application (`npm run tauri:dev`). Verify that the UI now displays the data generated by `MockMcpClient` (which includes some randomized values and alerts) and that it updates periodically.
    *   Update unit tests for `DefaultDashboardService` if necessary to handle the `McpClient` dependency (mocking the client).

**Phase 2: Real `McpClient` Implementation**

1.  **Create Real `McpClient`:** Implement a struct that conforms to the `McpClient` trait defined in `dashboard-core/src/mcp.rs`. This struct will contain a real MCP client instance (from the `crates/mcp`) and implement the trait methods (`get_metrics`, `get_protocol_data`, `get_alerts`, `acknowledge_alert`) by sending appropriate MCP commands/requests and parsing responses.
2.  **Inject Real Client:** Update the instantiation of `DefaultDashboardService` (likely where the Tauri app is built or where services are initialized) to inject this new, real `McpClient` implementation instead of the `MockMcpClient`.

**Alternative Strategy (Push-based via `update_dashboard_data`)**

*   Create an MCP Adapter/Listener: A separate component listens for relevant MCP events/messages (e.g., monitoring updates, protocol status changes, new alerts).
*   Call Service Method: When the adapter receives an update, it constructs a complete `DashboardData` object and calls `dashboard_service.update_dashboard_data(new_data).await`.
*   Simplify `collect_dashboard_data`: This function might become unnecessary or only handle basic system metrics not covered by MCP pushes. The `start` loop might not be needed if updates are purely event-driven. This aligns better with the existence of the `update_dashboard_data` method but requires setting up the MCP listening component.

**Decision:** Proceed with the **Polling via McpClient** strategy first, using `MockMcpClient` initially.
